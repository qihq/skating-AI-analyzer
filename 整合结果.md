# 花样滑冰动作分析系统

## 一、系统架构与技术栈

### 前端（Frontend）

- **框架**：React + TypeScript
- **状态管理**：Zustand（轻量、无需 Provider 包裹）
- **样式方案**：Tailwind CSS
- **视频播放**：HTML5 `<video>` 元素
- **WebSocket 通信**：原生 WebSocket API（用于实时分析进度推送）
- **构建工具**：Vite

### 后端（Backend）

- **框架**：FastAPI（Python 3.11+）
- **异步任务**：Celery + Redis（broker）
- **数据库**：PostgreSQL（存储用户信息、分析记录、历史数据）
- **ORM**：SQLAlchemy 2.0（async）
- **缓存**：Redis（除 Celery broker 外，也用于缓存分析结果、用户会话等）
- **对象存储**：MinIO / S3 兼容（存储上传视频、分析报告、关键帧图片等静态资源）
- **实时通信**：WebSocket（FastAPI 原生支持，用于向前端推送分析进度与中间结果）

### AI 分析模块（可拆分为独立微服务，也可作为后端内部模块）

- **大模型调用**：通过 OpenAI / Anthropic / 本地部署模型 API 进行文本推理与多模态分析
- **计算机视觉**：
  - OpenCV（视频抽帧、图像预处理）
  - MediaPipe Pose / MMPose（姿态估计）
- **评分逻辑**：基于 ISU 评分规则的自定义评分引擎（Python 实现）

### 部署

- **容器化**：Docker + Docker Compose
- **反向代理**：Nginx（静态资源 + API 转发）
- **进程管理**：Gunicorn（FastAPI）+ Celery Worker

## 二、系统核心功能模块

### 1. 视频上传与预处理模块

- **输入**：用户上传的滑冰视频（支持 MP4、MOV 等常见格式，建议 1080p+）
- **处理流程**：
  - 格式校验与时长检查（建议 30s ~ 5min）
  - 视频转码（统一为 H.264 编码，便于后续处理）
  - 关键帧提取（基于光流法或场景切换检测，提取动作起始/结束帧）
  - 视频分段（可选：自动检测单个动作边界，如跳跃的起跳→空中→落地）
- **输出**：标准化视频片段 + 关键帧图片序列

### 2. 姿态估计与动作捕捉模块

- **技术方案**：
  - 使用 MediaPipe Pose 或 MMPose 进行人体关键点检测（全身 33 个关键点）
  - 逐帧提取关节坐标（x, y, z + 置信度）
  - 基于关键点序列计算运动学参数：关节角度、角速度、身体重心轨迹、腾空高度等
- **数据结构**：每个时间帧对应一个 PoseFrame 对象，包含所有关键点坐标及衍生参数
- **输出**：时间序列化的姿态数据（JSON/protobuf 格式），供后续分析使用

### 3. 动作识别与分类模块

- **目标**：自动识别视频中的滑冰动作类型
- **支持的动作类别**（按 ISU 标准）：
  - **跳跃（Jumps）**：Axel、Salchow、Toe Loop、Loop、Flip、Lutz（区分单跳/双周/三周/四周）
  - **旋转（Spins）**：直立旋转、蹲转、燕式旋转、组合旋转
  - **步法（Step Sequences）**：直线步法、圆形步法、蛇形步法
- **实现方式**：
  - 基于规则的特征匹配（关节角度变化模式、腾空时长、旋转圈数等）
  - 可选：训练轻量级时序分类模型（如 LSTM / Transformer）进行动作分类
- **输出**：动作类型标签 + 动作时间区间 + 置信度

### 4. 质量评估与评分模块（核心）

#### 4.1 跳跃质量评估

- **起跳阶段**：
  - 起跳腿蹬伸角度与发力时机
  - 摆臂与浮腿协调性
  - 起跳前助滑速度与节奏
- **空中阶段**：
  - 旋转轴稳定性（通过重心轨迹偏移量衡量）
  - 身体收紧程度（手臂、腿部位置）
  - 旋转速度与周数是否匹配
- **落地阶段**：
  - 落冰腿屈膝缓冲角度（理想 120°~140°）
  - 落冰稳定性（重心偏移、是否单足落冰）
  - 浮腿展开姿态（是否自然、是否有额外支撑动作）
- **综合评分维度**：
  - 完成度（是否足周、是否双足落冰）
  - 技术执行分（GOE 加减分逻辑，参考 ISU 标准）
  - 流畅性与美感（主观维度，基于动作连贯性量化）

#### 4.2 旋转质量评估

- 中心稳定性（旋转轴是否偏移）
- 转速均匀性
- 姿态标准性（与标准姿态模板对比）
- 旋转进入与退出的流畅度

#### 4.3 步法质量评估

- 步伐节奏与音乐节拍匹配度（需结合音频分析，可选）
- 动作覆盖冰面范围
- 边刃使用准确性（内刃/外刃/平刃）
- 动作多样性与复杂度

### 5. 报告生成与可视化模块

- **分析报告内容**：
  - 动作识别结果总览（动作类型、时间轴、置信度）
  - 每个动作的详细评分（分维度打分 + 综合分）
  - 关键帧截图与姿态骨架叠加可视化
  - 改进建议（基于评估结果自动生成文字建议）
- **可视化形式**：
  - 时间轴视图（标注每个动作区间与评分）
  - 骨骼动画回放（可选：基于提取的姿态数据做 3D/2D 骨骼重放）
  - 雷达图（各维度能力对比）
  - 评分卡片（适合社交媒体分享）
- **输出格式**：网页端交互式报告 + PDF 导出（可选）

## 三、数据存储设计

### PostgreSQL 表结构（核心表）

```
users
├── id, username, email, password_hash, created_at

videos
├── id, user_id, file_path (S3/MinIO), original_filename, duration, resolution, upload_time, status (uploaded/processing/completed/failed)

analysis_sessions
├── id, video_id, user_id, status (pending/running/completed/failed), started_at, completed_at, overall_score

detected_actions
├── id, session_id, action_type (jump/spin/step), action_subtype (e.g., "triple_axel"), start_frame, end_frame, confidence, raw_score, goe_score

pose_frames
├── id, session_id, frame_number, timestamp_ms, keypoints_json (存储 33 个关键点坐标), derived_params_json

improvement_suggestions
├── id, session_id, action_id, category (technique/form/timing), suggestion_text, severity (info/warning/critical)
```

### Redis 缓存策略

- 分析任务进度：`analysis:{session_id}:progress` → 阶段 + 百分比
- 用户会话：`session:{user_id}` → JWT token / 会话信息
- 热门分析结果缓存：`result:{session_id}` → 序列化的报告数据（TTL 24h）

### 对象存储（MinIO/S3）目录结构

```
bucket/
├── videos/{user_id}/{video_id}/original.mp4
├── videos/{user_id}/{video_id}/processed.mp4
├── frames/{user_id}/{session_id}/frame_0001.jpg
├── reports/{user_id}/{session_id}/report.pdf
├── overlays/{user_id}/{session_id}/frame_0001_overlay.jpg  (骨架叠加图)
```

## 四、API 接口设计

### RESTful API

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/v1/auth/register` | 用户注册 |
| POST | `/api/v1/auth/login` | 用户登录，返回 JWT |
| POST | `/api/v1/videos/upload` | 上传视频（multipart/form-data） |
| GET | `/api/v1/videos/{video_id}` | 获取视频信息与状态 |
| POST | `/api/v1/analysis/start` | 提交分析任务（返回 session_id） |
| GET | `/api/v1/analysis/{session_id}` | 查询分析状态与结果 |
| GET | `/api/v1/analysis/{session_id}/actions` | 获取检测到的动作列表 |
| GET | `/api/v1/analysis/{session_id}/report` | 获取完整分析报告 |
| GET | `/api/v1/analysis/{session_id}/frames/{frame_id}` | 获取关键帧图片（含骨架叠加） |
| GET | `/api/v1/users/{user_id}/history` | 用户历史分析记录 |

### WebSocket 接口

- **连接地址**：`ws://host/ws/analysis/{session_id}`
- **推送消息格式**：
  ```json
  {
    "type": "progress",       // progress | action_detected | stage_update | error | completed
    "data": {
      "stage": "pose_estimation",
      "progress": 0.65,
      "message": "正在分析第 120/180 帧..."
    }
  }
  ```
- **消息类型**：
  - `progress`：分析进度更新（阶段 + 百分比）
  - `action_detected`：检测到新动作（实时推送给前端展示）
  - `stage_update`：分析阶段切换（如从"预处理"进入"姿态估计"）
  - `error`：分析过程中的错误信息
  - `completed`：分析完成，附带最终报告数据摘要

## 五、核心分析流程 Pipeline

```
用户上传视频
    │
    ▼
[1] 视频预处理（转码 + 关键帧提取 + 分段）
    │
    ▼
[2] 姿态估计（逐帧提取人体关键点）
    │
    ▼
[3] 动作识别（基于关键点序列识别动作类型与边界）
    │
    ▼
[4] 质量评估（按动作类型分别评估，生成分维度评分）
    │
    ▼
[5] 报告生成（汇总评分 + 生成建议 + 可视化渲染）
    │
    ▼
结果推送至前端（WebSocket 实时进度 + 最终报告）
```

**异步执行**：整个 Pipeline 由 Celery Worker 异步执行，前端通过 WebSocket 接收实时进度更新，避免长时间 HTTP 请求超时。

## 六、关键技术难点与解决方案

### 1. 跳跃周数判定

- **难点**：准确计算空中旋转角度，区分 2 周 / 3 周 / 4 周
- **方案**：基于肩髋分离角（shoulder-hip separation angle）的时间序列，结合旋转角速度积分，计算总旋转角度。辅以落地瞬间的身体朝向与起跳时的差值做校验。

### 2. 动作边界检测

- **难点**：连续滑冰动作之间没有明确的"暂停"信号
- **方案**：基于重心速度和关节角度变化率的突变检测，结合规则（如跳跃的腾空→落地模式）做动作切分。可使用滑动窗口 + 阈值检测或轻量级时序分割模型。

### 3. 评分主观性

- **难点**：滑冰评分具有主观性，难以完全量化
- **方案**：明确标注"技术维度评分"（可量化，如周数、落冰稳定性）与"艺术维度评分"（参考性，基于规则推算），并在报告中说明评分依据与局限性。初期以技术维度为主，艺术维度逐步迭代。

### 4. 视频质量依赖

- **难点**：低分辨率 / 远距离拍摄导致姿态估计不准
- **方案**：在预处理阶段进行视频质量评估（分辨率、模糊度、遮挡程度），若质量过低则提前提示用户并给出拍摄建议（距离、角度、光线等）。

## 重要修正：代码位置说明（必须先读）

`ai_skating_analysis_pack/` 是为了便于审查、整理出来的**代码快照与文档包**，**不是实际运行的源码目录**，Codex 不要在这个目录下做任何修改。所有改动必须落到 `skating-analyzer/` 仓库根目录下的真实工程位置。

### 路径映射（快照路径 → 实际工程路径）

由于实际目录结构在仓库内，Codex 必须先做以下两件事：

1. **第一步：定位实际代码**
   - 在仓库根目录运行 `grep -r "run_analysis_pipeline" --include="*.py" -l` 找到真正的 pipeline.py 实际路径（很可能在 `backend/app/services/` 或 `backend/app/analysis/` 之类目录）。
   - 类似地，用模块内的标志性函数名定位每个模块的真实位置：
     - `extract_motion_sampled_frames` → 实际的 video.py
     - `build_target_preview` → 实际的 target_lock.py
     - `extract_pose` → 实际的 pose.py
     - `infer_analysis_profile`、`JUMP_CHARACTERISTICS` → 实际的 action_profiles.py
     - `analyze_biomechanics`、`FPS = 5` → 实际的 biomechanics.py
     - `analyze_frames`、`normalize_vision_payload` → 实际的 vision.py
     - `generate_report`、`fuse_subscores` → 实际的 report.py
     - `smooth_phases` → 实际的 phase_smoother.py
     - `get_active_provider`、`request_text_completion` → 实际的 providers.py
     - `PIPELINE_VERSION` → 实际的 pipeline_version.py

2. **第二步：在 PR 描述中明确列出"快照路径 → 实际路径"对照表**，例如：
   ```
   ai_skating_analysis_pack/src/quality_assessment/biomechanics.py
     → backend/app/analysis/biomechanics.py  (实际路径，以 grep 结果为准)
   ```
   所有后续任务描述里出现的 `src/...` 路径，都按此对照表替换为真实路径再动手。
